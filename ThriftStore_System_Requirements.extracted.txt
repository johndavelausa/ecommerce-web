1. Project Overview
This document serves as the complete and authoritative specification for the Thrift Store E-Commerce Platform built using Laravel. It covers all functional requirements, user roles, business logic, UI/UX rules, and technical constraints. No additional features are needed beyond what is documented here.

4.2 Monthly Subscription Fee
There is a recurring monthly subscription fee of ₱500.

Sellers are notified when their subscription due date is approaching.
Admin approves or rejects subscription payments; approved payments reflect in Admin Total Profit.

Customer name and delivery address are automatically pre-filled from their profile.

5.4.1 Order Status Tracking

5.4.2 Marking Order as Received
Once the customer physically receives the product, they must update the order status to 'Order Received' or 'Completed'.
The order remains in 'Out for Delivery' status until the customer takes this action.
This action triggers a notification to the seller.
5.4.3 Order Cancellation
Customers can cancel their order within 30 minutes of placing it.
After 30 minutes, cancellation is no longer allowed.

Customers can view their complete order history.

Customers can leave ratings and written reviews on purchased products.

Customers can message the Seller of a product for inquiries.

6.1 Guest Capabilities
Guests can browse all publicly listed products.
Guests can send messages via the contact/inquiry form without an account.
Guest messages are visible to the Admin in the messages section.
Guests cannot add items to cart, purchase, or access any seller/admin functionality.
7.1 GCash Payment (Seller Fees Only)
GCash is used exclusively for seller-to-platform fee payments. The process is manual and requires Admin verification.
⚠ Note: There is NO automatic payment processing. All fee approvals are manual by the Admin. This applies to both Registration Fees and Monthly Subscription Fees.
7.2 Cash on Delivery (Customer Orders)
All customer purchases are handled via Cash on Delivery (COD).
No online payment gateway is integrated for customer purchases.
Payment collection happens physically when the order is delivered.
The wallet function is completely removed from the platform.
7.3 Payment History & Duplication Prevention
Complete payment history is maintained for both registration fees and subscription fees.
When displaying approved payment history, only approved transactions are shown.
The system must prevent duplicate payment records — if the same reference number is already recorded, the system must not allow it to be entered again.
8.1 No Redundancy Rule
If identical data already exists in the system, the system must not create a duplicate entry.
This applies to: products, orders, payments, user accounts, and messages.
Duplicate detection must be based on meaningful identifiers (e.g., GCash reference number, product name + price + seller).
8.2 No Automatic Payment Processing
The system must not auto-approve, auto-charge, or auto-process any payments.
All payments must be submitted manually by the user and approved/rejected manually by the Admin.
8.3 Order Tracking Number
When an order is marked as Shipped, a random tracking number is automatically generated and assigned.
This tracking number is visible to both the customer and the seller.
8.4 Delivery Confirmation Flow
An order cannot move to 'Delivered' status unless the customer explicitly marks it as received.
Until then, the order remains in 'Out for Delivery' / 'Shipped' status.
This ensures accurate delivery tracking and prevents premature status changes.

9.2 Responsive Design
The platform must be accessible and functional on both desktop and mobile browsers.
Forms, tables, and product grids must adapt to screen size.
9.3 System Branding
The Admin can change the platform logo via System Settings.
The Admin can update the background design/visual theme.
Changes take effect platform-wide immediately after saving.
10.1 Role-Based Access Control (RBAC)
Laravel middleware must enforce role-based routing for Admin, Seller, Customer, and Guest.
Each dashboard and route must be protected by role-specific guards.

Seeders for Admin user and system settings defaults.
